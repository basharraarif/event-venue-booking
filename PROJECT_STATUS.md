# Project Title: Event Booking Platform Backend

## Current Status
The backend for the Event Booking Platform is substantially implemented, providing core functionalities for user management, venue listing, event creation and management, and a booking system. All features are exposed via a RESTful API. API documentation is auto-generated and available. The system is configured for development with SQLite, console email, and local memory caching, with notes for production setup.

## Implemented Applications
The backend is organized into the following Django applications:

-   **`core`**: Handles user management, including a custom User model with additional fields (phone number, address) and user authentication endpoints.
-   **`venues`**: Manages venue information, including details like address, capacity, amenities, and pricing. Supports filtering and searching.
-   **`events`**: Manages event creation and details, including event name, description, associated venue, organizer, categories, start/end times, ticket price, and status. Supports filtering.
-   **`bookings`**: Manages the booking process, allowing users to book tickets for events. It handles booking status, number of tickets, and calculates total price. Implements user-specific views and admin oversight.

## Key Features
-   **User Authentication & Management**: Custom User model, registration, login/logout, password management via `dj_rest_auth` and `django-allauth`.
-   **Venue Management**: CRUD operations for venues, with filtering by capacity, amenities, etc., and text search.
-   **Event Creation & Management**: CRUD operations for events, linking to venues and organizers, categorization, status updates, and ticket pricing. Supports filtering by various criteria.
-   **Booking System**: Users can book events; their bookings are tracked with status and calculated total price. Users manage their own bookings; admins can manage all.
-   **API Documentation**: Auto-generated OpenAPI schema (via `drf-spectacular`) available through Swagger UI and ReDoc.
-   **Admin Interface**: Django Admin is configured for all key models (`User`, `Venue`, `Event`, `Category`, `Booking`) with customizations for better usability.

## Getting Started / How to Run

1.  **Dependencies**:
    -   Python dependencies are managed with Poetry.
    -   Navigate to the `event_booking_platform_backend` directory.
    -   Install dependencies: `poetry install`
    -   Activate the virtual environment: `poetry shell`

2.  **Database**:
    -   The project is configured to use SQLite for development. The database file (`db.sqlite3`) will be created in the `event_booking_platform_backend` directory.
    -   Run migrations to set up the database schema:
        ```bash
        cd event_booking_platform_backend
        python manage.py migrate
        ```

3.  **Environment Variables**:
    -   Create a `.env` file in the `event_booking_platform_backend` directory.
    -   Refer to `event_booking_platform_backend/.env.example` for required and optional variables.
    -   **`SECRET_KEY`**: This is mandatory for Django and must be set to a strong, unique secret string in your `.env` file.
        Example: `SECRET_KEY=your_strong_secret_key_here`
    -   **`DEBUG`**: Set to `True` for development (e.g., `DEBUG=True`). Defaults to `False`.

4.  **Running the Development Server**:
    -   Navigate to the `event_booking_platform_backend` directory.
    -   Run the server:
        ```bash
        python manage.py runserver
        ```
    -   The API will typically be available at `http://127.0.0.1:8000/`.

## API Documentation
The OpenAPI schema is automatically generated by `drf-spectacular`. You can access the interactive API documentation via:
-   **Swagger UI**: `http://127.0.0.1:8000/api/schema/swagger-ui/`
-   **ReDoc**: `http://127.0.0.1:8000/api/schema/redoc/`
-   The schema file itself can be generated using: `python manage.py spectacular --file schema.yml`

## Configuration Notes

-   **`SECRET_KEY`**: Must be set in a `.env` file located in the `event_booking_platform_backend` directory. A default placeholder is in `settings.py` but should not be used for production.
-   **`DEBUG` Mode**: Configurable via the `DEBUG` variable in the `.env` file (e.g., `DEBUG=True`). Defaults to `False` if not set.
-   **Database**:
    -   Currently uses SQLite for development (see `DATABASES` in `settings.py`).
    -   For production, PostgreSQL is recommended. Comments in `settings.py` provide guidance on configuring it (e.g., using `DATABASE_URL` environment variable and installing `psycopg2-binary` or `psycopg2`).
-   **Email**:
    -   Currently configured to output emails to the console (`EMAIL_BACKEND = 'django.core.mail.backends.console.EmailBackend'`).
    -   For production, this needs to be changed to a proper email backend (e.g., SMTP) with appropriate credentials set via environment variables.
-   **Caching**:
    -   Uses Django's local memory cache (`LocMemCache`) by default.
    -   Redis configuration is present (commented out) in `settings.py` and can be enabled by setting a `REDIS_URL` environment variable. Requires `django-redis`.

## Key Implemented Features (Details)
Building upon the core functionalities, several key features have been fully implemented and tested:
- **Stripe Payment Integration**: Secure payment processing for bookings via Stripe, including PaymentIntent workflow, webhook handling for payment confirmations and failures, and relevant email notifications.
- **Comprehensive Email System**: Automated email notifications for critical user actions: new user registration, booking confirmations (paid and free events), booking pending payment, booking cancellations, payment success, and payment failures. Uses Django's templating system for email bodies.
- **Advanced Role-Based Access Control (RBAC)**: Distinct user roles ('Customer', 'Event Organizer', 'Venue Manager', 'Admin') with tailored permissions for accessing and managing different parts of the platform (venues, events, bookings).
- **Ticket Price Snapshotting**: Ensures that the price of tickets at the time of booking is preserved, so subsequent changes to an event's price do not affect existing bookings.
- **Event Capacity Enforcement**: Robust checks to prevent overbooking, ensuring bookings do not exceed the defined capacity for an event (or its venue if event-specific capacity is not set).

## Known Limitations / Areas for Immediate Future Work

-   **Payment Integration**: Stripe payment processing is implemented for bookings, including PaymentIntent creation, webhook handling for status updates (success/failure), and related email notifications. Frontend integration with Stripe Elements is also in place.
-   **Email Notifications**: Email notifications for key events such as new user registration, booking confirmation (including for free and paid events), booking pending payment, booking cancellation, and payment success/failure are implemented and utilize templated emails.
-   **Frontend Integration**: The API is generic. Specific optimizations or endpoints tailored for a known frontend application have not been developed.
-   **Advanced User Roles/Permissions**: Granular roles (e.g., 'Customer', 'Event Organizer', 'Venue Manager', 'Admin') with specific permissions for managing venues, events, and bookings are implemented and enforced throughout the API.
-   **Event Capacity Management (High Concurrency)**: The current booking system robustly validates against event capacity. For very high concurrency scenarios, further hardening (e.g., database-level locking, already partially implemented with `select_for_update`) could be an area for future optimization if performance profiling indicates a need.
-   **Ticket Price Consistency**: The system snapshots the `price_per_ticket_at_booking` to ensure that changes to an event's ticket price do not affect the `total_price` of existing bookings.

## Production Configuration Notes
- **Media Files (User Uploads)**: For production environments, it is critical to configure external cloud storage (e.g., AWS S3, Google Cloud Storage via `django-storages` or similar) for handling user-uploaded media files (e.g., event images, venue photos). The default local storage (`MEDIA_ROOT`) is not suitable for production containerized deployments due to ephemeral file systems. Ensure `MEDIA_URL` and `MEDIA_ROOT` related settings, along with necessary cloud storage credentials (e.g., `AWS_STORAGE_BUCKET_NAME`, `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_S3_REGION_NAME`, `AWS_S3_ENDPOINT_URL` for S3-compatible services) are correctly configured via environment variables.
- **Database, Cache, Email**: Refer to the `Configuration Notes` section for guidance on setting up PostgreSQL, Redis, and a production-grade email backend.
